# MongoDB Connection Guidance for ByteDoc

Use this guide when generating or reviewing ByteDoc SDK connection code, diagnosing connection failures, or deciding whether a connection problem is caused by credentials, network boundaries, or runtime environment.

This guide adapts MongoDB official `mongodb-connection` agent-skill guidance for ByteDoc. See `../references/mongodb-upstream-adaptation.md` for included and excluded upstream skills.

## ByteDoc First

- Run `bytedcli --json bytedoc sdk plan --service <psm> --access-env <env> --language go` before suggesting a connection string.
- Treat `supportStatus`, `accessMethod`, `prerequisites`, `warnings`, and `references` as the source of truth.
- Do not invent hostnames, tokens, Consul endpoints, Mesh sockets, or direct MongoDB URIs.
- If `supportStatus` is `unsupported` or `needs_input`, stop and show `questions[]`, `warnings[]`, and `references[]` instead of generating code.

## Connection Code Rules

- For Go code, use ByteDoc official driver `code.byted.org/bytedoc/mongo-go-driver`; do not directly depend on upstream `go.mongodb.org/mongo-driver` for generated ByteDoc samples.
- Treat upstream MongoDB driver docs as API-shape reference only; ByteDoc package selection follows ByteDoc official SDK docs.
- Reuse one `mongo.Client` per process or component lifecycle; do not connect and disconnect per request.
- Use `context.WithTimeout` for `Connect`, `Ping`, and each query operation.
- Keep `SEC_TOKEN_STRING`, temporary usernames/passwords, and full URIs out of logs, errors, metrics, and panic output.
- Use `net/url` or equivalent escaping for temporary credentials; never interpolate raw passwords into a URI.
- Put close/disconnect logic at process shutdown or owner lifecycle boundaries, not in hot query paths.

## ByteDoc Access Methods

- `consul-token`: use runtime token injection such as `SEC_TOKEN_STRING`; do not ask the user to paste tokens.
- `consul-temporary-credential`: use temporary development credentials only for the supported local/dev environment returned by `sdk plan`.
- `mesh-token` / `mesh-socket`: confirm TCE/FaaS Mesh egress prerequisites before blaming driver code.
- `visual-query`: do not generate SDK code; guide the user to visualization/query workflows.

## Troubleshooting Flow

1. Re-run `bytedcli --json bytedoc sdk plan --service <psm> --access-env <env> --language go`.
2. Confirm the runtime matches the plan (`local-mac`, `devbox`, `tce`, or `faas`).
3. Check authorization using `bytedoc access permission get` or follow `BYTEDOC_ACCESS_REQUIRED.error.details.setup_commands`.
4. Confirm the code uses the generated access method and required env vars from `sdk generate`.
5. Only after ByteDoc facts are consistent, inspect driver-level errors, timeouts, pool usage, and context deadlines.

## Do Not

- Do not recommend Atlas connection setup, Atlas UI steps, or MongoDB MCP setup.
- Do not tune pool size as the first fix; gather latency, concurrency, and timeout evidence first.
- Do not suggest bypassing ByteDoc network boundaries with SSH tunnels or direct database endpoints.
